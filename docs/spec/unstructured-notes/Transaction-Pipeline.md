Transaction Pipeline ideation.

# 1. Transaction - informational flow
Basic output of Transact() or Onboard() methods, they trigger pipeline assessment within the target Product which could lead to various outcomes:
    - Transaction-Intents: instruct Transact() call to upstream provider, done in an aggregated manner, more than one upstrea provider can be triggered by single incoming Transaction-Intent, potentially with a future effective date
    - Fees: depending on the fee configuration setup of the Product a fee can be calculated and set into the fee application pipeline
    - Postings: depending on the configuration a particular posting may be triggered, potentially with a future value date
    - Asset transfers: depending on the configuration a particular asset transfer may be initiated, potentially with a future value date

Transaction-Intents are recorded in the log for each interaction and are stored in their unaggregated form for the duration of debug window setting. SQLite or another form to be considered as volume can be significant, and keeping everything in-memory can be prohibitive.

Each Product will have description (or rather assignment from transaction-pipeline configs) of Transaction-Inten pipeline as follows:
    - [transaction-intent-id]: incoming Transaction-Intent that can be called by clients
        - Destinations:
            - [agent-id-1]: describe outgoing aggregated Transact() to be sent to the agent-id-1
                - Transaction-Intent: Transact-Purchase-Clearing-Upstream
                - Value Date: same-day, next-working-day + X, next-day + X
                - Amount: [transaction-intent-amount] (formulas to be made available in the future)
                - Currency: [currency-code] / [fx-rate] if specified the transaction-intent amount will be converted using fx-rate specified
            - local: transaction-intent to be recorded locally and no further action to be taken 

# 2. Postings - accounting records

Postings operate on a structure of Ledger. Ledger is a Vendor-level property, but operates as a multi-Product, multi-currency record keeping vehicle with aggregation capacity. For example the Ledger could look like this:
    - Revenue: aggregate level revenue
        - [product-id] 
            - Fees: account used for fees collected in a particular product
    - Costs: aggregate level cost
        - [product-id] 
            -Costs: total product level costs
                - [agent-id] 
                    - Costs: costs accrued from a particular counterparty agent
    - Capital: aggregate total own funds
        - Accrued profits and losses: recorded profits and losses
        - Prefunding: funds deployed as cashflow gap buffers
        - Equity: starting capital  
    - Managed-Funds: total money held on behalf of others
        - [product-id] 
            - Customer-Funds: money held on behalf of customers of particular product
            - Settlement-Funds:
                [agent-id] -> ValueContainer([Managed-Funds][Settlement-Funds][agent-id])
                    - Outgoing: money due to others
                    - Incoming: money due from others

Ledger is described as part of vendor agent configuration. 

Subsequently in the Product level configuration the posting rules are defined with structures like this:
    - [transaction-intent-id] (corresponding to Transact-Purchase-Clearing-Upstream)
        - Source: [Managed-Funds][product-id][Customer-Funds]
        - Destination: [Managed-Funds][product-id][Settlement-Funds][upstream-agent-id]
        - Amount: [transaction-intent-amount] (formulas to be made available in the future)
        - Value Date: same-day, next-working-day + X, next-day + X, next-month-day + X
    - [fee-id]: for postings generated by fees

At least the above level logic shall be supported by GenericProduct, and more complicated logic may be further encapsulated in the specific products.

The posting is always made in a single currency between Deduct and Credit. All postings are dual-entry and must have Source/Destination specified.  

# 3. Asset transfers - value movement

Assets are transferred between value containers. Such value containers exist on Agent level (in the future they'd be tagged to VendorAgents who actually hold them on behalf of the Agent, but we start with all value containers to be local to the agent themselves - not how it works in the real world), and are defined as part of agent initialization in the scenario such as:
    - Own-Funds:
        - Operational-Account
        - Prefunding-Account:
            - [product-id]
    - Managed-Funds:
        - Customer-Funds:
            - [product-id]
        - Settlement-Funds:
            - [agent-id]

One could see that value containers somewhat follow the Ledger setup, but they aren't the same. Ledger are just a record of what assets and liabilities are there, but value containers have the actual assets (or liabilities, if the balance is negative, provided this is permitted for the container - this is configured on a per-container). 

There should also be a mapping table of Ledger entries to Value Containers for reconciliation purposes. For example in our scenario Settlement-Funds containers are mapped on aggregate basis as shown in the markup.

The value transfers are described in a similar to postings logic for both fees and transact-intents:
    - [transaction-intent-id] (corresponding to Transact-Purchase-Clearing-Upstream - generated from Transact-Purchase-Clearing)
        - Source: [Managed-Funds][Customer-Funds][product-id]
        - Destination: [Managed-Funds][Settlement-Funds][upstream-agent-id]
        - Amount: [transaction-intent-amount] (formulas to be made available in the future)
        - Value Date: same-day, next-working-day + X, next-day + X, next-month-day + X

# 4. Fees

Fees are configured in the fee-economics section and are at this stage driven by transaction-intents or other fees (hence fees are calculated in sequence specified). Then the fee models can be borrowed by individual products in their construction:
    - [sequence]
        - [fee-id]: the one being calculated
            - [transaction-intent-id] or [fee-id]: the one that causes fee to apply
                - Filter: [transaction-details-filter]: as future versions would add transaction-details (e.g. whether purchase is card-present, international, etc without having to create new transaction-intent as those aren't necessarily relevant for postings, etc)
                - Count-Cost: money-object that will be multiplied by the count of transaction-intents
                - Amount-Percentage: %% to be applied to the transaction-intent-amount
                - Formula: more flexible (in-line eval of Python code?) that allows for implementation of custom logic

The fees, once calculated, are recorded in the log, and are used to generate postings and value transfers. They are executed within [sequence], that is all fees in [sequence] get calculated and executed along with their postings/value transfers, then next [sequence] is applied that is potentially triggered by the fees calculated in prior sequence.

Detailed fee logs persist for the duration of debug window, similar to transaction-intents, and beyond that only aggregates remain. The same applies to value tranfers and postings: daily balances of each container (end of day) persist forever, and individual postings only as far as the debug window stretches.
